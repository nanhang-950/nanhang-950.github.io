tty_struct 基础知识#

tty_struct 是 kernel pwn 中非常有价值的数据结构。我们可以通过利用这个数据结构来达到提权的目的。

tty_struct结构#

struct tty_struct {
    const struct tty_operations *ops;  // ★ 虚表，可控即代码执行

    struct tty_driver *driver;
    int index;

    struct tty_bufhead buf;            // read/write 缓冲区

    unsigned long flags;
    ...                                 // 许多成员
};

首先我们查看 tty_struct 的源码。实际上，我并不了解太多 TTY 的内部逻辑，但只要知道该结构如何被使用，就足够进行利用。我们主要关注 /dev/ptmx 的初始化与 ioctl 实现。

初始化与 open#

/dev/ptmx 字符设备在 /drivers/tty/pty.c 中初始化：

static void __init unix98_pty_init(void)
{
  ...
  tty_default_fops(&ptmx_fops);
  ptmx_fops.open = ptmx_open;

  cdev_init(&ptmx_cdev, &ptmx_fops);
  if (cdev_add(&ptmx_cdev, MKDEV(TTYAUX_MAJOR, 2), 1) ||
     register_chrdev_region(MKDEV(TTYAUX_MAJOR, 2), 1, "/dev/ptmx") < 0)
  panic("Couldn't register /dev/ptmx driver");
  device_create(tty_class, NULL, MKDEV(TTYAUX_MAJOR, 2), NULL, "ptmx");
}

对应的 file_operations 与 tty_fops 几乎完全一致，只是 .open 变成了 ptmx_open。 因此，当你 open("/dev/ptmx") 时，会调用 ptmx_open()：

static int ptmx_open(struct inode *inode, struct file *filp)
{
  struct tty_struct *tty;
  int retval;
  ...
  retval = tty_alloc_file(filp);
  ...
  tty = tty_init_dev(ptm_driver, index);
  ...
  tty_add_file(tty, filp);
  ...
}

这就是相关结构初始化的地方。

tty_file_private 的分配#

首先 tty_alloc_file() 会通过 kmalloc() 分配一个 struct tty_file_private，并将其设置为 file->private_data：

int tty_alloc_file(struct file *file)
{
  struct tty_file_private *priv;
  priv = kmalloc(sizeof(*priv), GFP_KERNEL);
  ...
  file->private_data = priv;
  return 0;
}
  • struct tty_file_private 大小为 0x20，因此使用 kmalloc-32。
  • 接着调用 tty_init_dev()，其内部会初始化一个 struct tty_struct 并返回。

tty_struct 的大小为 0x2B8，因此使用 kmalloc-1024。其中重要成员如下：

struct tty_struct {
  int magic;                // 魔数校验
  struct kref kref;         // 引用计数
  struct device *dev;
  struct tty_driver *driver;// 必须有效
  const struct tty_operations *ops;  // vtable
  ...
}

tty_add_file#

tty_struct 分配完成后，tty_add_file() 被调用：

void tty_add_file(struct tty_struct *tty, struct file *file)
{
  struct tty_file_private *priv = file->private_data;
  priv->tty = tty;
  priv->file = file;
  ...
}

它将 tty_struct 的指针写入 tty_file_private.tty。

这一点非常关键，因为通过读取 tty_file_private，你可以泄露出 tty_struct 的堆地址。

ioctl#

当你对 /dev/ptmx 调用 ioctl() 时，会执行 tty_ioctl()。其中，首先会通过 file 找到对应的 tty_struct：

struct tty_struct *tty = file_tty(file);

实现如下：

static inline struct tty_struct *file_tty(struct file *file)
{
  return ((struct tty_file_private *)file->private_data)->tty;
}

接下来会执行 “paranoia check” 来检查 tty_struct.magic 是否为 0x5401：

若不匹配，tty_ioctl() 会直接返回。

随后进入 switch，如果找不到对应的 ioctl：

default:
  if (tty->ops->ioctl) {
    retval = tty->ops->ioctl(tty, cmd, arg);
    if (retval != -ENOIOCTLCMD)
    return retval;
  }

也就是说：

它会调用 tty->ops->ioctl，这是可以被我们利用的控制流劫持点。

你能做什么#

核心利用能力分两类：

  1. kheap 泄漏（堆地址 + kbase 泄漏）
  2. RIP 劫持（利用 kROP）

kheap 泄漏#

假设你有以下条件：

  • kmalloc-32 UAF
  • kmalloc-1024 UAF
  • 对 kmalloc-1024 有写原语（全写或部分写）

则可以：kmalloc-32 UAF → 泄漏 kmalloc-1024 地址

因为：

  • tty_file_private（32 字节）包含 tty_struct 的地址
  • tty_struct 在 kmalloc-1024

kmalloc-1024 UAF → 泄漏 kbase

因为：

  • tty_struct.ops 指向内核中的 ptmx_fops
  • 从而间接泄漏内核基址

基于 tty_struct 的 RIP 劫持#

这是核心部分，也是为什么该结构对 kernel pwn 极其宝贵。

假设：

你能从 tty_struct 头部连续写至少 0x200 字节。

那么你可以通过覆盖 tty_struct.ops 来获取 RIP。 注意：

  • 不能破坏 magic = 0x5401
  • 否则 tty_ioctl() 会拒绝继续执行

为什么需要 “两阶段 ROP”

在启用 KPTI 的系统上，你不能直接跳到用户态 payload。 因此需要：

  1. 在内核空间进行 stack pivot（修改 RSP）
  2. 进入稳定 ROP 链

而普通内核中几乎不会出现 mov rsp, <controlled> 这种 gadget。

于是我们利用 tty_struct 自身来做 “预 ROP（pre-ROP）”。

完整布局示例

struct tty_struct {
  +000: magic, kref : 0x5401
  +008: dev;        : pop rsp gadget（作为 leave 的返回地址）
  +010: driver;     : &tty_struct + 0x170（pivot 目标，必须为有效地址）
  +018: ops;        : &tty_struct + 0x50
  +020: index;
  ...
  +050: fake vtable（size=0x120，包含 leave gadget）
  ...
  +170: 实际 ROP 链
  ...
}

执行路径#

  1. tty_ioctl() → jmp 到 fake vtable，执行 leave
  2. leave：
    • mov rsp, rbp（rsp 指向 tty_struct 顶部）
    • pop rbp → rbp = magic|kref
  3. 返回地址位于 dev 字段 → pop rsp
  4. pop rsp 将 RSP 设置为 driver
  5. RSP 指向 +0x170 → 执行真正的 ROP 链

安全研究者的自我修养

2025年8月31日

前言#

在阅读了泉哥关于一次安全分享的总结之后感触良多，所以打算根据分享人的挖洞学习思路给自己制定一个学习训练计划。

原文链接：安全研究者的自我修养

根据研究方向选择了 Linux Kernel 为目标，按照分享人的思路以代码审计和 Fuzzing 复现历史的漏洞，在对 Linux Kernel 的历史漏洞进行了一些了解之后发现近来所有漏洞大都在网络子系统和驱动中产生，所以选择了几个相关漏洞进行复现复现分析：

读代码要带有目的去读,是要挖漏洞还是想要了解这个程序到在底干了些什么要理解整体的时候,千万不要在某个细节里钻牛角尖要一边理解细节一边挖洞的时候，记得要联想到所有可能的情况。

训练挖洞的双技能：

  1. 看洞：哪里看？历史漏洞的 git log、bug 报告、代码质量报告等等。
  2. 识洞：就是肉眼看代码找漏洞，即代码审计，难点也就是在这上面，训练方法继续往下看。

代码审计训练：

  1. 根据自己目标定位，寻找相应的历史漏洞案例进行学习，比如要搞 chrome 就找 chrome 的历史漏洞；
  2. 掌握漏洞所在的模块或子系统，但不看完整的漏洞细节描述，尝试在漏洞版本中找出对应的漏洞；
  3. 如果 2 中未能找出漏洞，就去看漏洞细节描述，对比自己的审计过程，看遗漏了哪一步骤；
  4. 不断重复上述训练，直至相信：挖洞只是体力消耗，而非能力问题。

Fuzzing 训练：

  1. 拿已公开的历史漏洞问自己：如何写 fuzzer 挖掘到此漏洞？
  2. 如果自己不知道此漏洞，那又能够挖掘到呢？
  3. 不断重复训练并改进 fuzzer，相信会有更多漏洞被意外发现。

漏洞信息#

CVE ID：CVE-2021-22555

由于 Linux Netfilter 模块在实现IPT_SO_SET_REPLACE（或IP6T_SO_SET_REPLACE）setsockopt时存在堆越界写入漏洞，导致攻击者可以利用漏洞实现权限提升。

在net/netfilter/x_tables.c中发现自 v2.6.19-rc1 以来影响 Linux 的堆越界写入。这允许攻击者通过用户名空间获得权限或导致 DoS（通过堆内存损坏）。

Linux 内核系统调用兼容性。Syzkaller 挖掘这个漏洞为什么不会那么明显？这个漏洞只涉及到创建socket和设置socket系统调用。

当以兼容模式调用 IPT_SO_SET_REPLACE 或 IP6T_SO_SET_REPLACE 时（该操作需要 CAP_NET_ADMIN 权限，但在用户+网络命名空间中可以获取），系统需要将结构体从用户态转换为内核态，同时也要从 32 位转换为 64 位，以便被本地函数处理。显然，这一过程容易出错。

我们的漏洞存在于xt_compat_target_from_user()函数中，其中memset()调用时使用了一个偏移量 target->targetsize，但该偏移量在内存分配时并未考虑进去 —— 导致有几个字节被写入到了分配范围之外，即发生了越界写。

// https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/net/netfilter/x_tables.c
void xt_compat_target_from_user(struct xt_entry_target *t, void **dstptr,
				unsigned int *size)
{
	const struct xt_target *target = t->u.kernel.target;
	struct compat_xt_entry_target *ct = (struct compat_xt_entry_target *)t;
	int pad, off = xt_compat_target_offset(target);
	u_int16_t tsize = ct->u.user.target_size;
	char name[sizeof(t->u.user.name)];

	t = *dstptr;
	memcpy(t, ct, sizeof(*ct));
	if (target->compat_from_user)
		target->compat_from_user(t->data, ct->data);
	else
		memcpy(t->data, ct->data, tsize - sizeof(*ct));
	pad = XT_ALIGN(target->targetsize) - target->targetsize;
	if (pad > 0)
		memset(t->data + target->targetsize, 0, pad);

	tsize += off;
	t->u.user.target_size = tsize;
	strlcpy(name, target->name, sizeof(name));
	module_put(target->me);
	strncpy(t->u.user.name, name, sizeof(t->u.user.name));

	*size += off;
	*dstptr += tsize;
}

targetsize 本身并不受用户控制，但用户可以通过指定不同名称的 target（如 TCPMSS、TTL 或 NFQUEUE）来选择结构体大小不同的目标。targetsize 越大，我们能够控制的偏移范围也就越大。

不过，target 的大小不能是 8 字节对齐的，只有当 pad > 0 时漏洞才能触发。

我找到的最大可用 target 是 NFLOG，它允许我们实现最多 0x4C 字节 的越界写。用户可以通过在 struct xt_entry_match 和 struct xt_entry_target 之间添加填充（padding）来进一步影响这个偏移。

struct xt_nflog_info {
	/* 'len' will be used iff you set XT_NFLOG_F_COPY_LEN in flags */
	__u32	len;
	__u16	group;
	__u16	threshold;
	__u16	flags;
	__u16	pad;
	char		prefix[64];
};

请注意，缓冲区的目的地分配了gfp_kernel_account，也可以在大小上有所不同：

// https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/net/netfilter/x_tables.c
struct xt_table_info *xt_alloc_table_info(unsigned int size)
{
	struct xt_table_info *info = NULL;
	size_t sz = sizeof(*info) + size;

	if (sz < sizeof(*info) || sz >= XT_MAX_TABLE_SIZE)
		return NULL;

	info = kvmalloc(sz, GFP_KERNEL_ACCOUNT);
	if (!info)
		return NULL;

	memset(info, 0, sizeof(*info));
	info->size = size;
	return info;
}

不过，该对象的最小大小大于 0x100（即 256 字节），这意味着它只能被分配到 kmalloc-512 或更大的 slab 中。换句话说，想要利用这个漏洞，我们必须寻找分配在 kmalloc-512 到 kmalloc-8192 之间的受害对象，以便让越界写覆盖这些对象的数据。

源码解析#

继 BleedingTooth（我第一次研究 Linux 的契机）之后，我也想找到一个提权漏洞。我从一些旧漏洞入手，比如 CVE-2016-3134 和 CVE-2016-4997，它们启发我在 Netfilter 代码中使用 grep 搜索 memcpy() 和 memset()，最终让我发现了一些有问题的代码。

  • translate_compat_table->xt_alloc_table_info，为新规则创建存储空间，用于将32位规则存储为64位规则
  • xt_compat_target_from_user存储完旧规则后，为target->targetsize对应的xt_target添加字节对齐pad，但该内容在创建存储空间阶段未考虑(用户定义xt_entry_target.u.user部分，后续会关联填充xt_entry_target.u.kernel，target->targetsize对应kernel)

跟踪调试，查看空间填充过程#

  • translate_compat_table->xt_alloc_table_info创建空间，查看&newinfo->entries偏移
  • compat_copy_entry_from_user->memcpy: *dstptr 偏移
  • compat_copy_entry_from_user->xt_compat_match_from_user: *dstptr 偏移
  • xt_compat_target_from_user: 查看xt_target *target，发现target->targetsize非0

后言#

挖掘一个目标的漏洞，首先我们要了解目标的架构、功能、可能的漏洞点，针对于几万行代码的小项目可以短时间分析尝试挖掘。

但是针对于操作系统、浏览器等这些比较大的项目我们就必须选择一个方向进行专向研究。

Fuzzing 和源码挖洞是相辅相成的，不要把能不能挖到漏洞的锅都丢给Fuzzing，Fuzzing 不出来就是没有漏洞；也不要把全部的精力都花在阅读源码上，有很多时候会迷失在代码里，忘记上下文到底在做些什么，越看越迷茫。这是成本与收益的博弈，对于代码量较大的程序来说，偏向 Fuzzing 的投入产出比较高；对于代码量较小的程序来说，偏向阅读源码的投入产出比较高。

漏洞信息#

漏洞描述：CVE-2024-1086 是一个影响 Linux 内核的高风险权限提升漏洞。由于 Netfilter 模块的 nf_tables 子系统中存在释放后重用（Use-After-Free）问题，这个漏洞允许本地经过身份验证的攻击者（包括具有低权限的用户）利用该缺陷提升至root权限。该漏洞的CVSS评分为7.8，并于2024年3月28日被发现。目前，漏洞的概念验证（PoC）和利用代码（EXP）已在互联网上公开。

漏洞概述#

根本原因： 在 nft_verdict_init() 函数 (位于 net/netfilter/nf_tables_api.c:9814) 中存在一个输入净化（input sanitization）错误，该错误允许规则判断（rule verdict）返回正的丢弃错误（positive drop errors）。此问题被分类为 CVE-2024-1086。

此漏洞导致了 struct sk_buff 对象以及 sk_buff->head 对象（kmalloc 对象，大小范围从 256 到 65536 字节，假设 IPv4，即 order 4 的 buddy pages）的稳定双重释放（double-free）原语。

接下来我们就一个文件一个文件的分析。

但是根据漏洞信息我们可以先想一下，UAF 漏洞的特征是什么？

  • 对象被释放后，内核仍然继续使用这个内存区域进行读写操作；
  • 跟哪些内存释放函数有关。

那么内核中的内存分配和释放函数是什么呢？

函数 功能 上下文/特点
kmalloc(size, flags) 分配内核小块内存（对象、缓冲区） 类似用户态 malloc()；flags 如 GFP_KERNEL（可睡眠）、GFP_ATOMIC（中断上下文）
kzalloc(size, flags) 分配并清零内存 等价于 kmalloc() + memset(..., 0, size)
vmalloc(size) 分配大块连续虚拟内存（不保证物理连续） 用于大缓冲区或数组，效率比 kmalloc 低
vzalloc(size) 分配大块虚拟内存并清零 等价于 vmalloc() + 清零

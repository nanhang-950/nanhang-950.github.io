简介#

CodeQL 是 GitHub 开发的一种静态分析工具，广泛用于代码安全性和质量的检测。CodeQL 允许开发人员使用一种类似于 SQL 的查询语言来分析源代码，并查找潜在的漏洞或代码缺陷。它通过创建一个“代码数据库”，使得用户可以查询程序中的各种数据流、控制流、API 调用等信息，从而发现安全漏洞、代码重复或不良编码实践。

CodeQL 是一种基于规则匹配的代码审计工具。使用一种名为 CodeQL 的查询语言，这种语言类似于 SQL，允许开发人员编写自定义的查询来分析代码的结构、数据流和控制流。

CodeQL 支持多种编程语言，包括但不限于：

  • C/C++
  • Java
  • JavaScript/TypeScript
  • Python
  • Go
  • Ruby
  • C#

本文主要研究的是 CodeQL C/C++ 的分析审计。

创建数据库#

您需要为项目创建一个 CodeQL 数据库，以便进行查询。可以使用codeql database create命令来创建。

codeql database create ./test-db --language=cpp --command="gcc test.c -o test"
  • test-db：指定数据库创建成功后存放在test-db目录下
  • language：指定语言，这里选择cpp即C++
  • command：编译指令，如果使用make编译那么就将make语句写在这里。

QL查询语句#

CodeQL 所使用的查询语言是一种类 SQL 的语法。

用于查找每种受支持语言最相关和最有趣的问题。您还可以编写自定义查询来查找与您自己的项目相关的问题。

重要的查询类型包括：

  • 警报查询：突出显示代码中特定位置的问题的查询。
  • 路径查询：描述代码中源和接收器之间的信息流的查询。

  • where子句

该子句定义要应用于子句中声明的变量以生成结果的逻辑条件。此子句使用聚合、谓词和逻辑公式将感兴趣的变量限制为满足定义条件的较小集合。codeql 库对特定语言和框架的常用谓词进行分组。

  • select子句

该子句指定要为满足子句中声明的变量显示的结果。select 子句的有效结构由元数据中指定的属性定义。

警报查询的select 子句由两个列组成，其结构如下：

select element,string
  • element：由查询标识的代码元素，用于定义警报的显示位置
  • string：一条消息，其中还可以包含连接和占位符，说明生成警报的原因

全局数据流分析#

import cpp
import semmle.code.cpp.dataflow.new.DataFlow

class EnvironmentToFileConfiguration extends DataFlow::Configuration {
  EnvironmentToFileConfiguration() { this = "EnvironmentToFileConfiguration" }
  override predicate isSource(DataFlow::Node source) {
    exists(Function getenv |
      source.asIndirectExpr(1).(FunctionCall).getTarget() = getenv and
      getenv.hasGlobalName("getenv")
    )
  }

  override predicate isSink(DataFlow::Node sink) {
    exists(FunctionCall fc |
      sink.asIndirectExpr(1) = fc.getArgument(0) and
      fc.getTarget().hasGlobalName("fopen")
    )
  }
}

from
  Expr getenv, Expr fopen, EnvironmentToFileConfiguration config, DataFlow::Node source,
  DataFlow::Node sink
where
  source.asIndirectExpr(1) = getenv and
  sink.asIndirectExpr(1) = fopen and
  config.hasFlow(source, sink)
select getenv, fopen

import cpp
import semmle.code.cpp.dataflow.TaintTracking
import DataFlow::PathGraph

class NetworkByteSwap extends Expr{
    NetworkByteSwap()    {
        exists(MacroInvocation mi |
            mi.getMacroName().regexpMatch("ntoh(s|l|ll)") and
            this = mi.getExpr()
          )
    }
}

class Config extends TaintTracking::Configuration{
    Config() { this = "NetworkToMemFuncLength"}

    override predicate isSource(DataFlow::Node source){
        source.asExpr() instanceof NetworkByteSwap
    }

    override predicate isSink(DataFlow::Node sink){
        exists(FunctionCall c | c.getTarget().getName() = "memcpy" and sink.asExpr() = c.getArgument(2))
    }
}

from Config cfg, DataFlow::PathNode source, DataFlow::PathNode sink
where cfg.hasFlowPath(source, sink)
select source, sink, "Network byte swap flows to memcpy"

数据流分析概述#

数据流分析 通过追踪程序中数据的流动来帮助识别潜在的漏洞，尤其是涉及不可信输入的漏洞，如 污染分析。在 CodeQL 中，数据流分析可以应用于局部数据流和全局数据流。

局部数据流#

局部数据流分析集中在单个函数内的数据流，它比全局数据流更快速、更精确。主要的组件有：

  • DataFlow::Node：表示数据流经的单元。节点可以是表达式节点或参数节点。

  • DataFlow::localFlow：跟踪节点间数据流的谓词，用于函数内部的局部数据流。

例如，以下 CodeQL 查询追踪从函数参数到 fopen 调用参数的数据流：

import cpp
import semmle.code.cpp.dataflow.new.DataFlow

from Function fopen, FunctionCall fc, Parameter p, DataFlow::Node source, DataFlow::Node sink
where
  fopen.hasGlobalName("fopen") and
  fc.getTarget() = fopen and
  source.asParameter(1) = p and
  sink.asIndirectExpr(1) = fc.getArgument(0) and
  DataFlow::localFlow(source, sink)
select p

该查询找到作为参数传递给 fopen 的 参数。

局部污染追踪#

污染追踪 扩展了局部数据流分析，通过追踪数据流经的不保值的操作（例如 用户输入）。例如，用户输入被污染，然后传递给 malloc。

使用 TaintTracking::localTaint 谓词来追踪污染传播，它与 localFlow 类似，但还包括不保值的步骤。

全局数据流#

全局数据流分析跟踪整个程序中的数据流，尽管它比局部数据流更强大，但精度较低，分析通常需要更多的时间和内存。

全局数据流通过实现 DataFlow::ConfigSig 配置来完成，配置中包括：

  • isSource：定义数据的来源。

  • isSink：定义数据的去向。

  • isBarrier：可选地限制数据流。

使用 DataFlow::Global 模块进行全局分析。例如：

module MyFlowConfiguration implements DataFlow::ConfigSig {
  predicate isSource(DataFlow::Node source) { ... }
  predicate isSink(DataFlow::Node sink) { ... }
}
module MyFlow = DataFlow::Global<MyFlowConfiguration>;

该配置允许你分析从源到汇的全局数据流。

全局污染追踪#

全局污染追踪是全局数据流的扩展，跟踪污染数据如何传播到程序中的关键点，如文件打开函数。类似于局部污染追踪，全局污染追踪有助于识别数据（例如来自不可信来源的数据）如何传播到文件操作等关键点。

例如，以下配置可以跟踪从环境变量到 fopen 的数据流：

module EnvironmentToFileConfiguration implements DataFlow::ConfigSig {
  predicate isSource(DataFlow::Node source) {
    exists(Function getenv | source.asIndirectExpr(1).(FunctionCall).getTarget() = getenv)
  }
  predicate isSink(DataFlow::Node sink) {
    exists(FunctionCall fc | sink.asIndirectExpr(1) = fc.getArgument(0) and fc.getTarget().hasGlobalName("fopen"))
  }
}
module EnvironmentToFileFlow = DataFlow::Global<EnvironmentToFileConfiguration>;
